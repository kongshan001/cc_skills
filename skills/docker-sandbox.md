# Docker Sandbox

## 1. 背景需求

AI 代理在执行任务时经常需要运行代码、安装包或探索未知环境，这些操作可能存在风险：
- 运行不可信代码导致系统受损
- 实验性包污染本地环境
- 多个任务之间的依赖冲突

需要一个安全隔离的沙箱环境来执行这类操作。

## 2. 目标

创建一个与主机隔离的 Docker 容器环境，让 AI 代理可以在其中安全地：
- 运行任何代码和命令
- 安装和探索软件包
- 执行临时任务而不影响主机

支持 Claude、Codex、Copilot、Gemini、Kiro 等主流 AI 代理。

## 3. 设计方案

**核心架构**：
- 基于 Docker 容器提供完整的 Linux 环境隔离
- 支持网络代理控制，防止意外外联
- 自动清理机制，避免资源泄漏
- 多代理共享沙箱池管理

**能力矩阵**：
| 功能 | 支持情况 |
|------|----------|
| 基础命令执行 | ✅ |
| 包安装（apt/npm/pip等） | ✅ |
| 网络访问控制 | ✅ |
| 文件系统持久化 | 可选 |
| GPU 加速 | 需要特殊配置 |
| 多容器并行 | ✅ |

## 4. 本地部署

```bash
clawhub install docker-sandbox
```

**依赖**：
- Docker CLI 和守护进程
- 网络访问（拉取基础镜像）

**快速验证**：
```bash
docker ps  # 确认 Docker 运行
docker run --rm hello-world  # 验证基本功能
```

## 5. 效果展示

触发示例：
- "在沙箱中运行这段代码"
- "用 npm 安装这个包测试一下"
- "探索一下这个 Python 项目的依赖"

执行效果：AI 在隔离容器中执行命令，返回结果后可选择保留或销毁环境。

## 6. 优缺点分析

| 优点 | 缺点 |
|------|------|
| 完全隔离，零风险 | 启动有延迟（秒级） |
| 环境一致性好 | 镜像体积较大 |
| 支持多种 AI 代理 | 需要 Docker 权限 |
| 自动清理，资源可控 | 不适合长时间运行的任务 |

## 7. 平替对比

| 方案 | 特点 | 适用场景 |
|------|------|----------|
| Docker Sandbox（当前） | 完整隔离，支持多代理 | 安全执行代码 |
| Docker Essentials | 本地直接操作，无隔离 | 已知安全的操作 |
| 本地虚拟环境 | 轻量但隔离性弱 | Python 隔离 |
| 容器直接执行 | 无额外抽象 | 熟悉 Docker 的用户 |

## 8. 落地过程

**推荐安装评估**：
- **本地开发机** ⭐⭐⭐⭐⭐ — AI 代码执行必备
- **ECS 云服务器** ⭐⭐⭐⭐⭐ — 隔离测试环境，避免污染生产

**使用场景**：
1. **代码探索**：运行未知来源的代码片段
2. **包测试**：安装新包而不污染本地环境
3. **多版本测试**：不同 Python/Node 版本的隔离测试
4. **安全研究**：处理可能有害的文件
